## 代码规范  
  
代码编写规范请参考阿里巴巴开发手册：https://github.com/alibaba/p3c  
  
1. 类名、变量名使用驼峰命名，不得使用下划线连接，名称命名尽量使用英文，如：UserInfo，不得使用拼音简写，如果使用拼音则需要使用全拼，  
   如：YongHu（用户）,不得使用 Yh  
2. 日志一律打印到集成日志，不得打印到 ecology.log 等其它配置文件，方便日志管理，如果打印到 ecology.log 文件，会因文件太大而导致日志查看困难  
3. 不得忽略异常，不能直接打印异常，如：`e.printStackTrace()`,必需将异常打印到集成日志中，如：`log.error("错误", e)`  
4. 方法体不得过长，最好不超过60行，最好在30行内，太长的方法体不利于阅读，注意将不同功能拆分到不同方法中，不要所有功能都放到一个方法内。  
5. 避免体积过大的类，代码尽量在300行以内，遵循单一职责原则，按职责将功能划分到不同的类中。  
6. 对于公共方法，必需在方法上添加注释，包括方法的功能、参数说明、返回值说明和抛出的异常说明等。  
7. 每个类必需添加类注释，说明类的功能。  

## 接口规范

1. 二开的接口路径一律以 `/second-dev/` 开头（这里省略了 api 前缀，这是会默认加上的），如：`/second-dev/user/add`，如果是对外接口，则以 `/second-dev/open/` 开头，  
   如：`/second-dev/open/user/add`。  
2. 接口路径需要符合 RESTful 风格，路径按模块划分，如流程模块的接口：`/second-dev/workflow/` ，建模模块的接口：  
   `/second-dev/cube/`  

## 包路径规范

建议所有开发文件包路径都在 secdev 包之下，区分二开文件与标准文件，例如：`com.api.interfaces.secdev.extend.web`

在以下带有 secdev 的路径中，secdev 是建议的包路径。

- WEB 接口发布包路径：`com.api.interfaces.secdev.*`
- WEB 接口业务包路径：`com.engine.interfaces.secdev.*`
- 其它开发包路径：`com.customization.secdev.*`
  
## 开发包打包规范  
  
1. 建议将 class 文件打成 jar 包，这样容易打包，将你在本客户开发的所有 class 文件都打到 jar 包中，jar 包命名示例：`seconddev-sgmw-yll.jar` , seconddev 表示这是二开的 jar 包，sgmw 为客户名称，yll 为你姓名的缩写。
2. 开发包内如果存在后端文件，需要放在开发包的 ecology 文件夹内，将 class 文件放在 ecology/classbean 中，如果有其它后端文件也需要放在 ecology 文件夹内，方便部署到服务器上，对于配置文件可单独放在另一个文件夹中，如放在 /配置文件/ecology/WEB-INF/props/ 中。  
3. 如有需求文档，需要放在开发包内，如 /需求文档/xxx.docx  
4. 开发包内文件要求：
	- 说明文档（必需）：必需包含作者（如果是外部开发人员，需注明机构名称），时间，开发的功能（描述实现了什么）、使用方法（如需要，不涉及使用操作可忽略）、如何部署和配置
	- 需求文档（可选）：原始的需求文档，这样有利于追溯原始需求和源头
	- 接口文档（可选）：如果开发的接口包含复杂的请求参数（如请求体参数超过5个），且参数需要在系统中配置（如在 ESB 中），或者供外部系统调用，则需要编写接口文档，说明接口的请求参数、返回参数等信息，接口文档可以是在线的或者离线，如果是在线文档将文档地址说明在开发说明文档中，如果是离线的文档 则需要放在开发包内。  
5. 如果开发需要其它附加文件，如果 ecode 应用、建模（导出zip文件）、ESB应用等，都需要放到开发包内。  
  
开发包目录结构参考：  
  
```  
根目录  
|-ecology  
|   |-classbean  
|   |-WEB-INF  
|-配置文件  
|   |-ecology  
|       |-WEB-INF  
|           |-props  
|               |-dev.properties  
|-ecode  
|-建模  
|-需求文档  
|-  |-需求.docx  
|-开发说明.docx  
```
